# Research — 조사 및 작업 명세 (파이프라인 1/4단계)

당신은 이 단계에서 **플래너 겸 조사자**입니다. 아래 요청을 코드베이스 근거에 기반해 분석하고, 다음 단계(`/implement`)가 추가 질문 없이 바로 실행할 수 있는 작업 명세를 작성하세요.

요청:

$ARGUMENTS

오늘 날짜: !`date +%Y-%m-%d` (비어 있으면 Bash로 `date +%Y-%m-%d`를 직접 실행해 확인)

## 절차

### 1. 작업 폴더 결정
- 경로 형식: `docs/tasks/<오늘 날짜>-<slug>/` (slug는 요청을 요약한 영문 소문자-하이픈, 예: `docs/tasks/2026-06-10-query-execution-api/`)
- 이 폴더에 최종 산출물 `research.md` 하나만 생성합니다.

### 2. 탐색 및 식별
- Glob, Grep, Read 도구로 관련 파일·함수·컴포넌트·설정을 찾습니다.
- 탐색 범위가 넓거나 여러 영역에 걸치면 **Agent 도구로 Explore 서브에이전트를 병렬 실행**해 요약만 회수하세요. 메인 컨텍스트를 파일 덤프로 오염시키지 않는 것이 목적입니다.

### 3. 심층 분석
- 식별된 파일은 **반드시 직접 읽고** 확인합니다. 관련 타입 정의, import 구조, 호출 관계까지 추적합니다.
- 수정 대상에 의존하는 다른 파일을 추적해 부작용(side-effect) 후보를 파악합니다 (영향도 분석).

### 4. 명세 작성 (계약 수립)
- 요청이 암묵적으로 요구하는 에러 처리, 엣지 케이스, 통합 지점까지 명세에 포함하세요 (과소 산정 방지).
- 단, 명세는 **고수준으로 유지**합니다. 실제 구현 코드를 미리 작성하지 마세요 — 세부 기술 결정을 여기서 과도하게 못 박으면 오류가 후속 단계로 연쇄됩니다. "어떤 논리로 무엇을 바꾸는지"까지만 명시합니다.
- **성공 기준(Acceptance Criteria)** 은 이 파이프라인의 계약입니다. `/implement`는 이것을 충족해야 하고 `/qa`는 이것으로 채점합니다. 각 기준은 반드시 "어떻게 확인하는가"가 포함된 검증 가능한 문장이어야 합니다.
- 고수준 유지와 검증 가능성의 경계: AC는 **외부에서 관찰 가능한 동작·출력 계약**을 고정하고, 내부 구현 선택(키 이름, 자료구조, 포맷 세부 등)은 "구현 재량"으로 명시해 미해결 질문 또는 Action Items에 남깁니다.
- **모바일 AC (UI 작업 필수)**: 요청이 frontend UI(레이아웃·컴포넌트·페이지)를 변경하면, AC에 모바일 뷰포트 기준을 **반드시 1개 이상** 포함하세요 (CLAUDE.md "모바일 대응" 참조. 예: "375px 뷰포트에서 가로 스크롤·요소 겹침·잘림 없음 — **/qa 단계에서** 브라우저 도구로 확인". 브라우저 E2E 확인 방법을 AC에 쓸 때는 수행 주체가 /qa임을 명시해, /implement가 자기 일로 해석하지 않게 합니다). UI를 전혀 건드리지 않는 작업이면 research.md에 "모바일 AC: 해당 없음 (UI 변경 없음)"을 명시해, 누락이 아니라 판단이었음을 기록합니다.

## 규칙 (Strict)

1. **코드 수정 금지**: 이 단계에서 Edit 도구는 차단되어 있습니다. Write 도구는 **오직 `research.md` 1개 파일에만** 사용하세요. Bash는 읽기 전용 명령(`git log`, `git blame`, `date` 등)에만 사용하고, 파일을 생성·수정하는 명령(리다이렉트 `>`, `mv`, `rm`, 패키지 설치 등)은 금지입니다.
2. **추측 금지**: 파일 내용을 직접 확인하지 않고 파일명·경험만으로 내부 로직을 단정하지 마세요. 확인하지 못한 사항은 본문에 쓰지 말고 "미해결 질문" 섹션에 기록하세요.
3. **근거 필수**: 본문의 모든 주장에는 `경로:줄번호` 형식의 근거를 답니다.
4. 범용 아키텍처 강의는 생략하고, 당면 요청 해결에 필요한 실용적 분석에 집중하세요.

## 산출물: research.md (고정 스키마)

아래 템플릿의 **모든 섹션을 채워서** `docs/tasks/<날짜>-<slug>/research.md`로 저장하세요.

```markdown
# Research: <제목>

- 날짜: YYYY-MM-DD
- 요청 원문: <사용자 요청 그대로>

## 요약
<현황·원인·해결 방향 3~7문장>

## 관련 파일 및 근거
- `경로:줄번호` — <이 파일/위치가 관련된 이유>

## 영향도
- <변경 시 영향받는 파일/모듈과 이유. 없으면 "없음 — 근거">

## 성공 기준 (Acceptance Criteria)
- [ ] AC-1: <검증 가능한 기준. 확인 방법 포함. 예: "X 함수에 빈 배열 입력 시 빈 결과를 반환한다 — 테스트 Y 실행으로 확인">
- [ ] AC-2: ...

## Action Items
- [ ] <원자적 작업 단위. 구현 코드가 아니라 변경 방향만>

## 미해결 질문
- <조사로 확정하지 못한 것, 사용자 결정이 필요한 것. 없으면 "없음">
```

## 완료 조건 (Definition of Done)

- [ ] `research.md`가 생성되었고 모든 섹션이 채워짐
- [ ] 모든 성공 기준에 확인 방법이 포함됨
- [ ] research.md 외에는 어떤 파일도 생성·수정하지 않음

완료 후 마지막 메시지에 작업 폴더 경로를 명시하고, 다음 단계로 `/implement docs/tasks/<날짜>-<slug>` 실행을 안내하세요. 미해결 질문이 있으면 함께 강조하세요.
