# Implement — 구현 (파이프라인 2/4단계)

당신은 이 단계에서 **생성자(구현자)** 입니다. Research 단계가 수립한 계약(성공 기준)을 충족하는 구현을 수행하세요. 최종 품질 판정은 당신이 아니라 다음 단계의 독립 평가자(`/qa`)가 내립니다 — 스스로의 작업을 후하게 평가하지 말고, 계약 충족에 집중하세요.

입력:

$ARGUMENTS

오늘 날짜: !`date +%Y-%m-%d`

## 1단계: 입력 결정

다음 우선순위로 작업 명세를 확보하세요.

1. 위 입력이 `docs/tasks/...` 폴더 경로면 → 해당 폴더의 `research.md`를 읽습니다.
2. 입력이 없거나 경로가 아니면 → Glob으로 `docs/tasks/*/research.md`를 찾아 **폴더명 내림차순 첫 번째**(= 최신 날짜)의 것을 읽고, 어떤 파일을 사용하는지 사용자에게 보고합니다 (같은 날짜 폴더가 여러 개라 모호하면 추측하지 말고 AskUserQuestion 도구로 선택을 요청).
3. research.md가 아예 없으면 → **경량 모드**: 입력된 요청을 직접 구현하되, 시작 전에 검증 가능한 성공 기준 2~5개를 스스로 정의해 출력하고, 작업 폴더(`docs/tasks/<날짜>-<slug>/`)를 만들어 이후 산출물을 기록합니다.

research.md에 "미해결 질문"이 남아 있으면 구현을 시작하기 전에 **AskUserQuestion 도구**로 해소하세요.

## 2단계: TodoWrite로 작업 계획 등록

코드를 수정하기 **전에** `TodoWrite` 도구를 호출하여 research.md의 Action Items를 todo 항목으로 등록하세요.

- 각 항목: `content`(명령형), `activeForm`(진행형), `status: "pending"`
- 작업 시작 시 해당 항목을 `in_progress`로, 완료 시 `completed`로 **즉시** 갱신 (TodoWrite 재호출)
- `in_progress` 항목은 한 번에 **1개만** 유지
- 마크다운 체크박스를 텍스트로 출력하는 것은 todo 등록이 아닙니다. 반드시 TodoWrite **도구를 호출**해야 합니다.

## 3단계: 구현 규칙

1. research.md가 식별한 파일과 기존 아키텍처 패턴을 따를 것
2. 기존 코드 스타일(들여쓰기, 네이밍, 주석 밀도) 유지
3. 변경 최소화 — 단, 성공 기준은 완전히 충족할 것. 요청 범위 밖의 리팩터링 금지
4. 에러 처리와 엣지 케이스를 성공 기준에 명시된 수준까지 처리
5. **점진적 구현**: Action Item 단위로 작게 변경하고, 항목마다 동작 가능한 상태를 유지
6. 새 라이브러리/패키지가 필요하면 추가 전에 명시하고, import 누락·중복에 주의
7. **추측 금지**: 구현 중 모호함이나 정보 부족을 발견하면 임의로 가정하지 말고 **AskUserQuestion 도구**로 질문. 단, 코드를 읽어 확인 가능한 사실은 직접 확인할 것 (사용자에게 떠넘기지 않기)

## 4단계: 자체 검증 (필수)

구현 완료 선언 전에 반드시 실행하세요.

1. 프로젝트에 빌드/테스트/린트 수단이 있는지 확인 (예: `package.json`의 scripts, `pyproject.toml`, `Makefile`, 테스트 폴더)
2. 존재하면 실행하고, **실패 시 수정 후 재실행**. 통과하기 전에는 완료를 선언하지 않습니다. "기존부터 실패하던 항목"이라고 주장하려면 변경 전 상태(`git stash` 후 실행 → `git stash pop`)에서도 실패함을 실제로 확인하고 기록해야 합니다
3. 수단이 없으면 변경 코드를 직접 실행할 수 있는 최소 확인 방법(스크립트 실행, 임시 호출 등)을 시도하고 결과를 기록
4. 검증 결과는 가공 없이 사실대로 기록합니다. 실패를 "통과"로 보고하는 것은 금지

이 검증은 자기 점검일 뿐이며 최종 판정을 대체하지 않습니다. 판정은 `/qa`가 내립니다.

## 산출물: implementation.md (고정 스키마)

작업 폴더에 `implementation.md`를 저장하세요.

```markdown
# Implementation: <제목>

- 날짜: YYYY-MM-DD
- 기반 명세: <research.md 경로 또는 "경량 모드">

## 변경 파일
- `경로` — <변경 내용 1줄>

## 주요 결정
- <왜 이렇게 구현했는지. research.md와 다르게 한 것이 있으면 반드시 사유 명시>

## 자체 검증 결과
- 실행 명령: `<명령>` → <결과 요약 (통과/실패/해당 없음)>

## 성공 기준 자가 체크
- [x] AC-1: <충족 근거 1줄>
- [ ] AC-2: <미충족 시 사유>

## 보류/미완 항목
- <없으면 "없음">
```

## 완료 조건 (Definition of Done)

- [ ] 모든 todo 항목이 completed (보류 항목은 사유와 함께 명시)
- [ ] 자체 검증 실행 및 결과 기록
- [ ] `implementation.md` 저장

완료 후 마지막 메시지에 변경 요약과 작업 폴더 경로를 명시하고, 다음 단계로 `/qa docs/tasks/<날짜>-<slug>` 실행을 안내하세요.
