## 요약

사용자 제안은 실제 데이터와 잘 맞는다. 저장소 루트의 뱅크샐러드 파일(이체 157행)을 직접 분석한 결과: ① **내계좌이체 36행 중 10쌍(20행)은 같은 날짜·같은 금액·반대 부호·다른 결제수단으로 자동 페어링 가능**해 한 건의 이체(출금→입금)로 무인 변환할 수 있다(예: 카카오페이 머니 → WON 통장). ② 미페어 내계좌이체 16행은 상대 계좌가 엑셀에 없는 계정(NH, 카카오뱅크 오픈뱅킹 등)이라 사용자 판단이 필요하다. ③ 내계좌이체 외 대분류(이체 62·카드대금 18·투자 17·저축 14·미분류 7·현금 3 = 121행)는 페어 가능 건이 0 — 전부 한쪽 다리만 기록돼 있어 사용자 제안대로 업로드 시점 분류가 적합하다.

따라서 엑셀 업로드를 **2단계(미리보기 → 확정)** 로 확장한다. 미리보기는 파일을 파싱만 하고(DB 무변경) 이체 타입 행을 "검토 대상"으로 반환하며, 행마다 기본 제안(페어된 내계좌이체 → 이체 자동, 카드대금 → 이체 권고, 그 외 → 부호 기반 수입/지출)을 붙인다. 사용자는 행별로 **수입 / 지출 / 이체(상대 계정 선택) / 건너뛰기** 중 결정하고, 확정 호출에서 일반 행과 함께 원자적으로 커밋된다. 월 단위 업로드 기준 검토 대상은 평균 약 13행(157행/12개월)으로 부담이 작다.

중요한 함정 하나: **카드대금 출금을 '지출'로 분류하면 이중 계산**이다 — 카드 개별 사용 내역이 이미 지출 행으로 존재하기 때문(`backend/app/excel_import.py:4-5`). 검토 UI는 카드대금 행에 이체(상대=카드 계정)를 기본 제안하고, 지출 선택 시 경고를 표시해야 한다.

## 영향도

- **선행 명세 전체가 전제**: transfer kind·상대 계정 컬럼·잔액 집계·수동 이체 입력 폼은 선행 명세대로 구현돼야 본 작업이 성립한다.
- **업로드 UX**: 기존 1클릭 업로드가 이체 행이 있는 파일에서는 2단계가 된다. 이체 행이 없는 월은 검토 단계 없이 기존과 동일 흐름이어야 한다(회귀 방지).
- **통계**: 수입/지출로 전환된 이체 행은 대시보드 수입/지출 합계에 새로 포함된다(기존에는 스킵이라 0). 이는 의도된 동작 변화 — 외부인 송금이 실제 수입/지출로 잡히게 됨.
- **재업로드**: 같은 월·구성원 재업로드 시 결정을 다시 입력해야 한다(결정은 저장되지 않음). 명세상 허용 — 미해결 질문에 개선 여지 기록.
- **카테고리 공간**: 수입/지출 전환 시 원본 대분류 기반 카테고리가 자동 생성될 수 있음(예: "이체 > 미분류" kind=expense). 기존 자동 생성 동작과 일관.

## 성공 기준 (Acceptance Criteria)

검증 환경: `docker compose up` 스택 + 저장소 루트의 실제 xlsx. 선행 명세의 AC-1~9(모델·집계·수동 입력·마이그레이션)는 **그대로 유효**하며, 선행 AC-10(이체 행 스킵 유지)은 아래로 대체된다.

- [ ] AC-1: 미리보기 호출(파일+월)은 DB를 변경하지 않고, 이체 타입 행 전부를 검토 대상으로 반환한다(행번호·날짜·원본 분류·내용·금액 부호·결제수단·기본 제안 포함) — 실제 xlsx로 호출 후 검토 목록과 거래 수 무변화 확인.
- [ ] AC-2: 같은 날짜·같은 금액·반대 부호·다른 결제수단의 내계좌이체 두 행은 한 건의 이체(출금→입금)로 자동 제안되고, 확정 시 두 행이 하나의 transfer 거래로 적재된다 — 실제 파일의 "카카오페이 머니→WON 통장" 쌍이 있는 월로 확인.
- [ ] AC-3: 검토 행별 결정이 확정에 반영된다: ① 이체(상대 계정 지정) → transfer 거래로 적재되어 두 계정 잔액에 ±반영, ② 수입/지출 → 해당 kind 거래로 적재되어 대시보드 합계에 포함, ③ 건너뛰기 → 미적재되고 결과 요약에 건수 표시 — API 호출 전후 `/analytics/assets`·`/analytics/dashboard`·`/transactions` 비교로 확인.
- [ ] AC-4: 이체 결정인데 상대 계정 누락 또는 출금=입금이면 확정이 422로 거부되고 어떤 행도 적재되지 않는다(원자성) — API 호출로 확인.
- [ ] AC-5: 이체 행이 없는 월의 업로드는 검토 단계 없이(또는 빈 검토 목록으로) 기존과 동일하게 완료된다 — 이체 행 없는 월로 확인.
- [ ] AC-6: 같은 월·구성원 재업로드 시 이전 확정에서 적재된 거래(transfer 포함, `source="import"`)가 교체되고 수동 입력 거래는 보존된다 — 두 번 업로드 후 거래 목록으로 확인.
- [ ] AC-7: 업로드 다이얼로그 검토 화면에서 행별로 수입/지출/이체/건너뛰기를 선택할 수 있고, 카드대금 행은 이체가 기본 제안되며 지출 선택 시 이중 계산 경고가 표시된다 — 브라우저(:3000)에서 확인.
- [ ] AC-8 (모바일): 375px 뷰포트에서 업로드·검토 다이얼로그에 가로 스크롤·요소 겹침·잘림이 없다 — 브라우저 도구로 375px 확인.
- [ ] AC-9: 수입/지출로 전환된 행은 원본 이체 분류를 추적할 수 있는 흔적(예: memo)이 남는다 — 전환 후 거래 상세로 확인. (표기 형식은 구현 재량)

## 미해결 질문

- **미페어 내계좌이체의 상대 계정이 앱에 없는 경우**(NH, 카카오뱅크 등 미추적 계좌): 검토 UI에서 신규 계정 즉석 생성을 허용할지, 기존 계정 선택만 허용하고 없으면 수입/지출/건너뛰기로 처리할지 — 사용자 결정 필요. 권고: v1은 기존 계정 선택만(계정은 자산 설정에서 미리 생성), 즉석 생성은 후속.

- 페어 자동 제안을 사용자가 해제(두 행을 각각 따로 분류)할 수 있게 할지 — 구현 재량, 권고: v1은 해제 가능하게 단순 토글.
