# Research: 엑셀(뱅샐현황) 업로드로 부동산·주식 평가액 자동 갱신

- 날짜: 2026-06-16
- 요청 원문: 자산 중에 부동산과 주식은 평가일 마다 금액을 갱신해야하는데 어떤 방법으로 하면 좋을까? → (사용자 결정) 수동 입력 UX 개선 및 엑셀 업로드 시 첫 번째 시트인 "뱅샐현황"의 부동산·주식 평가액으로 자동 갱신. 참조 파일: `2025-06-11~2026-06-11.xlsx`

## 요약

현재 부동산·주식(`real_estate`/`stock`) 계정의 평가액은 자산 페이지에서 **계정마다 다이얼로그를 열어 기준일·금액을 손으로 입력**하는 방식만 있다(`AssetsPage.tsx:358-460`, `valuations.py:22-42`). 평가 메커니즘(`AssetValuation`, `(account_id, date)` upsert)은 이미 완성돼 있고, 사용자가 느끼는 부담은 "평가일마다 일일이 갱신"하는 **운영 비용**이다.

해결 방향은 **기존 엑셀 업로드 플로우에 평가액 자동 반영을 끼워 넣는 것**이다. 뱅크샐러드 내보내기 파일의 첫 시트 "뱅샐현황" → "3.재무현황" 자산 표에는 `항목/상품명/금액` 형태로 부동산과 투자성 자산 평가액이 들어 있다(아래 근거). 업로드 확정 시 이 표를 파싱해, 상품명과 같은 이름의 계정을 찾아(없으면 해당 유형으로 자동 생성) **오늘 날짜의 `AssetValuation`으로 upsert**한다. 거래 가져오기(`transactions.py:148-385`)는 "가계부 내역" 시트만 보고 있으므로, 평가액 파싱은 별도 경로로 추가하되 같은 업로드 안에서 함께 처리한다.

사용자 확정 결정:
1. **반영 항목**: 부동산(`항목=부동산`) → `real_estate`, 투자성 자산(`항목=투자성 자산`) → `stock` (둘 다)
2. **계정 매칭**: 상품명으로 자동 매칭, 없으면 해당 유형으로 **자동 생성**
3. **평가일**: 업로드 시점의 **오늘 날짜**

### 엑셀 "뱅샐현황" 자산 표 구조 (참조 파일 직접 확인)
- 시트 목록: `['뱅샐현황', '가계부 내역']` — 첫 시트가 "뱅샐현황".
- "3.재무현황" 섹션 헤더가 행 35, 자산 표 헤더(행 38): `B='항목'`, `C='상품명'`(C:D 병합), `E='금액'`. (부채는 같은 행 F~I, 무관)
- 데이터 행 예시 (1-based, `항목 | 상품명 | 금액`):
  - 행 48: `투자성 자산 | SK텔레콤 | 19700000`
  - 행 49~57: 항목 셀이 **병합(B48:B57)되어 빈 값** — `기업은행/CMA계좌/MYW통장/금현물전용/기본계좌/우리사주예탁/위탁/종합매매…` (일부 금액 0)
  - 행 58: `부동산 | 오산시티자이2단지 | 355000000`
  - 행 59: `동산 | 기아 더 뉴K7 | 7890000` (반영 대상 아님)
  - 행 63: `총자산` (집계 행 — 반영 대상 아님)
- 함의: 파서는 **`항목` 셀을 마지막 비어있지 않은 값으로 carry-forward**해야 하고(병합 셀), `항목 ∈ {부동산, 투자성 자산}`인 행만 골라 `상품명+금액`을 읽는다. 행 번호는 파일마다 다를 수 있으므로 **하드코딩 금지 — `재무현황`/자산 헤더(`항목`,`상품명`,`금액`)를 탐지해 동적으로 위치를 잡는다**.

## 영향도

- **`excel_import.py`**: 신규 파서 함수 추가(기존 `parse_ledger` 미변경). 부작용 낮음.

## 미해결 질문

- **0원 항목 처리**: 투자성 자산에는 금액 0인 상품(`금현물전용`, `우리사주예탁`, `종합매매` 등)이 다수다. 모두 반영하면 자산 페이지 주식 그룹에 0원 계정이 다수 자동 생성돼 지저분해질 수 있다. 권장 기본값: **0원 행은 신규 계정을 만들지 않되, 이미 존재하는 매칭 계정이면 0으로 갱신**. (대안: 0원 전부 포함 / 전부 제외) — 구현 시 이 기본값으로 진행하되 사용자 확인 필요.
- **상품명-기존계정 유형 충돌**: 투자성 자산의 `기업은행` 등 상품명이 이미 다른 유형(예: `bank`) 계정으로 존재하면, 이름 매칭으로 그 계정에 평가액이 붙어 잔액 산정 방식이 바뀐다. 권장: 이름이 같으면 기존 계정 유형을 변경하지 않고 평가액만 부착(현 모델 동작). 충돌이 우려되면 매칭을 "이름+유형 동시 일치"로 좁히는 대안 — 사용자 결정 필요.
- **재업로드 시 사라진 항목**: 이전 업로드엔 있었으나 이번 파일엔 없는 상품의 과거 평가액은 그대로 둔다(삭제하지 않음). 정리가 필요하면 별도 정책 필요 — 현재 범위 밖.
- **평가일 타임존**: "오늘 날짜"는 백엔드 서버 로컬 `date.today()` 기준. 사용자 로컬과 다를 수 있으나 가계부 용도상 허용 가정.
- **수동 입력 UX 추가 개선**: 기존 평가 다이얼로그(수동 입력)는 그대로 유지하며, 이번 작업의 "UX 개선"은 엑셀 자동 반영으로 실현한다. 다이얼로그 자체의 추가 개선(여러 계정 일괄 입력 등)은 명시 요청 전까지 범위 밖.
- **참고**: 조사 중 참조 파일 검사를 위해 로컬에 `openpyxl` 패키지가 설치되었다(백엔드 의존성에 이미 포함). 코드/저장소 파일은 수정하지 않음.
